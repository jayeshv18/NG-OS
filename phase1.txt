now, our 64-bit CPU is trapped in a simulated cage called 16-bit Real Mode. Because the A20 memory wire is turned off by default, any memory address you try to generate that goes past the 1 Megabyte mark will automatically wrap back around to zero.
A20 Line (Address Line 20) is a physical electronic wire inside the computer's CPU that controls access to memory addresses above 1 Megabyte (MB).
it acts like a hardware lock that you must explicitly unlock before your OS can access modern RAM.
he original Intel 8086 processor was a 16-bit CPU. It had exactly 20 address wires (labeled A0 through A19).
With 20 wires, the maximum amount of RAM it could physically talk to was \(2^{20}\) bytes, which is exactly 1 Megabyte.
If a program tried to access memory slightly above 1 MB, the math would overflow, the 21st bit would drop off, and the address would wrap around back to the very beginning of RAM (0x0000). Old programs relied on this "wrap-around" behavior to work.

When IBM designed the next computer (the AT) using a faster chip that could read more RAM, they realized those old programs would crash because the memory wouldn't wrap around anymore.
To fix this without breaking old software, IBM added a physical hardware switch to the motherboard that forced the 21st wire (the A20 Line) to stay off by default, keeping the computer trapped inside 1 MB of memory for compatibility

When a modern x86_64 computer turns on, it starts up in 16-bit Real Mode, mimicking that exact same ancient environment. The A20 Line is locked shut.
If you try to switch your kernel into 32-bit Protected Mode or 64-bit Long Mode without unlocking the A20 line, your computer will only be able to see odd-numbered Megabytes of RAM (e.g., it can see MB 1, but MB 2 is invisible, MB 3 works, MB 4 is invisible).This causes immediate, chaotic memory corruption.

successfully unlocked the 21st memory wire. The CPU now has physical access to Gigabytes of RAM. But we are still in 16-bit mode
we want to switch the CPU to 32-bit Protected Mode, we have to destroy the concept of 16-bit segments. We do this by creating the Global Descriptor Table (GDT).
GDT is literally just an array of raw bytes stored in our assembly file. It tells the CPU hardware the new "rules" of memory. We need to define three specific entries (descriptors) in this table:
1. The Null Descriptor (8 bytes of zeroes): 8 bytes filled with absolute zero (0x00).
Intel hardware requires the very first entry in the GDT to be completely empty.
It serves as a guardrail. If we or a program make a mistake and try to read a pointer that equals 0 (a null pointer), the CPU looks at this first index in the GDT, sees that it is empty and invalid, and immediately stops execution. This stops a tiny bug from corrupting your whole system.
(If a program accidentally uses a null pointer (Segment 0), the hardware will see this empty descriptor and safely crash the program instead of overwriting memory.)

2. The Code Segment Descriptor:
A structure defining where software code sits and how it can execute.
This defines where executable code lives. To fake a flat memory model, we tell the CPU this segment starts at physical address 0 and spans the entire 4 Gigabytes of addressable space. We set the flags to mark it as Executable and Ring 0 (Kernel Privilege).
Base = 0, Limit = 4GB: we fool the CPU into thinking our code can live anywhere in the entire 32-bit space.
Executable Flag: Tells the hardware that the CPU is allowed to read these bytes as machine instructions (like mov or jmp).
Privilege Level (Ring 0): Sets the permission level to maximum authority. This is our kernel space.

3. The Data Descriptor:
This defines where variables and memory live. Exactly like the Code Segment, we set its base to 0 and limit to 4 Gigabytes. We set the flags to mark it as Read/Write, but NOT Executable (a security measure so hackers can't execute data).
A structure defining where variables, stacks, and data live.
Base = 0, Limit = 4GB: Exactly like the code descriptor, it maps the whole 4GB spectrum.
Read/Write Flag: The CPU is allowed to read from and modify these memory locations (essential for modifying variables).
Non-Executable (NX) Flag: This is a major hardware security feature. It tells the CPU: "we can read and write data here, but if someone tries to jump here and execute these bytes as code, trigger a crash immediately." This stops hackers from injecting malicious software commands into your variables.

We are building an array of raw numbers. To control it, we just need a few bookmarks (labels) to tell the assembler where things start and stop:
gdt_start: This marks index 0.
gdt_end: This marks the very finish line.
The Math: By subtracting gdt_end - gdt_start, the computer automatically calculates exactly how many bytes big our table is, no matter what we wrote inside.

Intel requires 8 bytes of zeroes.The Code: we use dd (Define Double-word = 4 bytes) two times.
4 bytes + 4 bytes = 8 bytes of pure zero. This is our safety trapdoor.

now the gdt is loaded bu the cpu is ignoring it, we have to flip the switch to turn it into the 32 bit processor.
->CR0 switch:
Inside the processor is Control Register 0 (cr0). The absolute first bit of this register (Bit 0) is the Protection Enable (PE) bit.
We cannot modify cr0 directly.
1. Move cr0 into a general-purpose register. (We can use eax, the 32-bit version of ax, because the CPU actually supports 32-bit registers in 16-bit mode if you force it).
2. Use the logical OR instruction to flip the first bit to 1 (which is 0x1).
3. Move eax back into cr0.
The instant that last instruction executes, the CPU is in 32-bit Protected Mode.

There is a massive hardware hazard here. The CPU pre-fetches instructions to run faster. Its internal pipeline is currently full of 16-bit instructions it decoded a microsecond ago. If it executes them now, the CPU will crash.
We must forcefully flush the pipeline. We do this by executing a Far Jump.
A standard jmp just moves the instruction pointer. A Far Jump updates both the Code Segment (CS) register AND the instruction pointer.
Because we are in Protected Mode now, the CS register no longer holds a mathematical segment. It holds an Offset pointing into our GDT.
Entry 1 is the Null Descriptor (Offset 0x0).
Entry 2 is the Code Segment (Offset 0x08 - because the Null descriptor is 8 bytes long).
Entry 3 is the Data Segment (Offset 0x10).

We must jump to the Code Segment (0x08) and absolutely cannot jump to the Data Segment (0x10) because of hardware-enforced security.
Look back at the Access Bytes we just wrote for the GDT:
    Code Segment Access Byte: 10011010b (The 4th bit from the right is 1, meaning Executable).
    Data Segment Access Byte: 10010010b (The 4th bit from the right is 0, meaning Not Executable).
When we execute a Far Jump, we are forcing a new value into the CS (Code Segment) register. The CPU's Instruction Pointer uses the CS register to fetch the next command.
Before the CPU runs the command, the hardware checks the GDT. If you try to jump to 0x10 (Data Segment), the CPU sees that the Executable bit is 0.
It instantly throws a hardware exception (a General Protection Fault). Because we haven't written an error handler yet, the CPU will simply panic, triple-fault, and reboot the machine.

The Hackers and the "W^X" Rule:
Intel design the hardware this way? To prevent malicious takeovers.
Imagine you are running a web server. The server receives data from the internet (like a URL string) and stores it in variables inside your RAM. That RAM is mapped to your Data Segment.
If a hacker sends a string that actually contains malicious compiled Assembly code (a classic Buffer Overflow attack), and the CPU was allowed to execute the Data Segment, the hacker would instantly take control of your computer.
By strictly separating Code (Read/Execute) from Data (Read/Write), the hardware ensures that even if a hacker injects malicious code into your variables, the CPU will physically refuse to execute it.


Once we land from the Far Jump, we are officially executing 32-bit code.
We must explicitly tell NASM to start assembling 32-bit instructions using the bits 32 directive.
We must update all the old 16-bit data segment registers (ds, ss, es, fs, gs) to point to our new 32-bit GDT Data Segment (Offset 0x10).
We must move the stack pointer to a safe 32-bit memory address (like 0x90000) so our C code will have plenty of room later.

Up until now, you have been compiling your bootloader using nasm -f bin.
The -f bin flag tells NASM to output a "Flat Binary." A flat binary is just a naked, raw stream of 1s and 0s. There are no headers, no metadata, and no labels. It is pure silicon logic.
This was perfect for Sector 1 because the motherboard's BIOS literally only accepts 512 bytes of naked machine code.
However, now we are trying to glue your Assembly file (stage2.asm) to your C file (kernel.c). We use a tool called a Linker (ld) to do this.
If you hand the Linker a naked Flat Binary, it will throw an error. It looks at the 1s and 0s and says, "I have no idea what this is. Where does the code start? Where are the variables? What are the function names?"

ELF (Executable and Linkable Format) fixes this.
When you compile using nasm -f elf32, NASM takes your machine code and wraps it in a highly structured envelope. This envelope includes a built-in dictionary. It explicitly tells the Linker:

"Here is where the executable code (.text) lives."
"Here is where the read-only data (.rodata) lives."
"I am looking for a function called kernel_main."

Because your C compiler (gcc) also outputs an ELF file, the Linker can read the dictionaries from both files, match up the missing pieces, and stitch them perfectly into one unified flat binary (kernel.bin).

1. The Time Bomb: Missing cli
There is a hardware clock on your motherboard that fires an interrupt exactly 18.2 times a second. Every time it fires, the CPU pauses what it is doing, jumps into the BIOS to update the system time, and comes back.
Look at your stage2.asm. You flipped the CPU into 32-bit Protected Mode (mov cr0, eax), which immediately revokes access to the 16-bit BIOS. But you never told the motherboard to stop firing interrupts!
Right as your loop was printing the letter 'l' in "loaded", the timer interrupt fired. The CPU tried to jump to the BIOS, realized it was in 32-bit mode with no Interrupt Descriptor Table (IDT), panicked, and instantly triple-faulted before QEMU could even render the rest of the text on your screen.
The Fix: We must explicitly clear (disable) interrupts before loading the GDT.

The motherboard has a Programmable Interval Timer (PIT) that fires a hardware
; interrupt 18.2 times every second. In 16-bit Real Mode, the BIOS catches this
; interrupt, updates the system clock, and returns control to our code.
;
; The moment we flip the PE bit in cr0 to enter 32-bit Protected Mode, we lose
; the BIOS entirely. If the timer ticks right as we are transitioning, the CPU
; will pause our code, look for a 32-bit Interrupt Descriptor Table (IDT) to
; handle the timer, realize the IDT doesn't exist yet, and instantly Panic/Triple-Fault.
; (This is why our print loop died mid-sentence at "Stage 2 l...").
;
; THE FIX: 'cli' (Clear Interrupt Flag)
; This instruction modifies the CPU's EFLAGS register, explicitly telling the
; processor to go deaf. It blocks all maskable hardware interrupts (like the
; timer and keyboard). We execute this right before loading the GDT to create
; a perfectly silent, safe vacuum for the CPU to transition into 32-bit mode.
; We will not use 'sti' (Set Interrupt Flag) to turn them back on until we
; have written our C kernel's IDT to safely handle them.

2. The Linker Padding Explosion
   Look at your linker.ld file. You have ALIGN(4K) for every section.
   This tells the linker: "Put the code here, and then pad the file with thousands of zeros until you hit the next 4,096-byte boundary before adding the next section."
   Because of this, your .rodata (which holds your C strings) and your .data (which holds your vga_buff pointer) are getting pushed so far back into the binary that they exceed the 15 sectors you are loading. The CPU reaches for the VGA pointer, grabs an empty 0x00000000 from unloaded RAM, and crashes.
   The Fix: Change ALIGN(4K) to ALIGN(4). This tightly packs your kernel so it easily fits in a tiny footprint.

To flush the CPU pipeline and enter 32-bit mode, we execute a Far Jump:
; 'jmp 0x08:init_pm'.
;
; Because this instruction is written inside our 'bits 16' zone, the Assembler
; normally compiles it assuming 'init_pm' is a small 16-bit memory address.
; However, because we are passing this file to an ELF32 Linker to connect it
; with our C Kernel, the Linker calculates 'init_pm' as a massive 32-bit address.
;
; Without the 'dword' keyword, the assembler truncates (chops off) the top half
; of the 32-bit address. The CPU jumps to the wrong location in memory, hits
; empty garbage data, and crashes silently.
;
; THE FIX: 'jmp dword 0x08:init_pm'
; Adding 'dword' (Double Word / 32 bits) forces the 16-bit assembler to allocate
; enough space for the full 32-bit memory address provided by the ELF Linker.
; This guarantees we land perfectly at the 'init_pm' label inside our C-linked RAM.

Hex Printer
When debugging an operating system, the computer's memory does not store letters like "A" or "B"; it stores raw numbers and bytes.
If a hardware error occurs, the CPU generates a numerical error code (like 0x0E for a Page Fault or 0x0D for a General Protection Fault).

Right now, our vga_print function is built specifically to print ASCII strings (text letters).
    If we tell your current driver to print the number 13, it will not print the digits "1" and "3" on your screen. It will look at the ASCII chart, see that 13 represents a "Carriage Return" control character, and print nothing at all.
    If a driver crashes and gives you a raw memory address error like 0x7C00, you cannot pass that number to vga_print. It expects text characters ending in \0.

The keyboard is a chaotic, interrupt-driven piece of hardware. When you press a key, it fires a raw numerical hardware scan code (e.g., pressing the 'Esc' key fires the raw number 0x01).
    If your keyboard code has a bug or maps a key incorrectly, you need to see exactly what number the hardware chip blasted to the CPU.
    Without a hex printer, you cannot see these raw incoming hardware numbers. You are coding completely blind.
A hex printer is a conversion function. It takes a raw numerical integer (like the number 255), mathematically tears it apart into individual 4-bit chunks (nibbles), translates those chunks into text characters ('F' and 'F'), and then sends those characters to your VGA screen as the string "0xFF".

->so in short the vga is designed to only read and display letters, it cannot read numbers directly, so we need a hex printer to see those raw data and translate for the vga to display and read and also if any error is there it says.
    VGA Limitation: The VGA hardware only knows how to look up character symbols (like 'A', 'B', or '1') from its built-in font grid. It cannot read a raw numerical integer value.
    The Translator: The hex printer acts as a translator. It takes a raw value like the number 10, converts it to the text character 'A', and gives that to the VGA.
    The Diagnostic Tool: This allows you to immediately read raw data from hardware chips, and crucially, see the exact error code if your system crashes.

1. Will this help us type numbers, letters, or both?
The hex printer function is strictly for output (printing things on the screen). It does not handle typing from the keyboard at all.
However, it helps you build the keyboard driver so you can eventually type both letters and numbers
The Problem: When you type the letter 'A' on a physical keyboard, the keyboard hardware sends the raw number 0x1E to the CPU. If you type '1', it sends 0x02.
The Help: The hex printer allows you to see these raw numbers on the screen while you are building your keyboard driver. Once your driver knows that 0x1E means 'A' and 0x02 means '1', you can map them, and you will be able to type both letters and numbers perfectly. If a bug causes a system crash during this process, the hex printer will display the exact error code.
